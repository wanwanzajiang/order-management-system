## 第三轮：修复RLS无限递归（致命Bug根因）

### 问题现象
用户报告登录时提示"获取用户信息失败"，所有账号都无法登录

### 根因分析
user_profiles表的RLS策略存在**自引用**导致PostgreSQL检测到 **infinite recursion**：
```sql
-- 这个策略查询了user_profiles自身 → 无限循环！
CREATE POLICY "admins_can_read_all_profiles" ON user_profiles FOR SELECT
USING (EXISTS (SELECT 1 FROM user_profiles WHERE id = auth.uid() AND role = 'admin'));
```
同样有问题的还有 `admins_can_update_any_profile` 和 `admins_can_insert_profile`

当auth.js登录后执行 `SUPABASE.from('user_profiles').select('*').eq('id', user.id).single()` 时触发递归报错

### 修复操作（直接在数据库执行）
1. 删除3个导致递归的策略
2. 新增2个安全策略：
   - `authenticated_can_read_profiles`: SELECT USING (auth.role()='authenticated') — 所有已登录用户可读
   - `users_can_insert_own_profile`: INSERT WITH CHECK (auth.uid()=id) — 仅可创建自己
3. 保留原有：`users_can_read_own_profile`(SELECT) + `users_can_update_own_profile`(UPDATE)

### 验证结果
- admin/warehouse/sales三个账号登录验证全部通过 ✅
- 查询user_profiles正常返回角色数据 ✅
- 查询orders表正常（空数组，无报错）✅
- SQL迁移文件同步更新防止以后重建再出现 ✅

### 经验教训
**绝对不要在表的RLS策略中用EXISTS子查询引用自身表！**这会导致infinite recursion。如需判断用户角色，应使用auth.jwt()或改用其他设计方式。
